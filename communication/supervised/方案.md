# 少量标注方案：特征原型分类

## 目标

在人工只标注极少样本的情况下，对 Agent trace 输出 `good case` 或 `bad case`。建议每类标注 1 到 5 个样本，总量很小也可以运行。

这个方案不把“生成大纲”或“报错信息”当作唯一判断条件，而是把每条 trace 转成行为特征向量，再比较待预测样本与 good/bad 标注样本的相似度。

## 输入格式

脚本会自动检测输入路径：

- 如果输入是单个 `.json` 文件，则该文件是一个样本，样本名为文件名。
- 如果输入是单个 `.jsonl` 文件，则每一行是一个样本；若该行对象中包含 `sample_name` 字段，则样本名为 `sample_name` 的值，否则样本名为行序号，例如 `1`、`2`、`3`。
- 如果输入是文件夹，则会读取其中的 `.json` 和 `.jsonl` 文件。
- 文件夹中的 `.json` 文件按“相对路径”作为样本名，例如 `a.json` 或 `sub/a.json`。
- 文件夹中的 `.jsonl` 文件同样优先使用行内 `sample_name` 字段；若没有该字段，则按“文件名:行序号”作为样本名，例如 `data.jsonl:1`、`sub/data.jsonl:3`。

行序号从 1 开始。空行会被跳过。若多个样本使用相同的 `sample_name`，标注匹配会变得不明确，建议上游保证 `sample_name` 唯一。

## 标注文件

标注文件格式为 CSV 或 TSV，至少包含两列：

```text
file,label
a.json,good case
data.jsonl:1,bad case
data.jsonl:2,good case
```

如果输入是单个 `.jsonl` 文件，标注里的 `file` 可以直接写行序号：

```text
file,label
1,good case
2,bad case
```

如果 JSONL 行里有 `sample_name` 字段，标注里的 `file` 应直接写该字段值：

```text
file,label
case_001,good case
case_002,bad case
```

`label` 支持以下写法：

```text
good case / good / 1
bad case / bad / 0
```

标注文件使用 `utf-8-sig` 编码读取，方便 Windows Excel 编辑。

## 指定参与评判的字段

可以通过 `--fields` 指定 trace 中参与 good/bad 判断的字段，只支持第一级字段，不支持 `a.b` 或 `a[0]` 这样的嵌套路径。

例如只使用 `query` 和 `plan_list`：

```powershell
python .\supervised\classify_supervised.py <trace路径> --labels labels.csv --fields query,plan_list
```

如果 JSONL 行中包含 `sample_name`，它仍然只用于样本命名和标注匹配，不受 `--fields` 影响。由于当前特征主要来自 `query` 和 `plan_list`，如果 `--fields` 不包含 `plan_list`，任务行为相关特征会变为空，通常不建议这样设置。

## 核心思路

1. 抽取行为特征。

   与无监督方案相同，包含任务数量、任务多样性、command 多样性、result 非空比例、result 长度、连续重复、循环模式、是否生成大纲、query 与任务名重合度等。

2. 使用标注样本做 robust normalization。

   不直接比较原始数值，避免某些大数值特征主导距离。

3. 构造 good 原型和 bad 原型。

   对 good 标注样本的特征求均值，得到 good 原型；对 bad 标注样本的特征求均值，得到 bad 原型。

4. 按距离分类。

   对每个待预测样本，分别计算它到 good 原型和 bad 原型的距离：

   ```text
   score = sigmoid(distance_to_bad - distance_to_good)
   ```

   score 越接近 1，越像 good case；越接近 0，越像 bad case。

5. 输出 nearest_good 和 nearest_bad。

   这有助于人工判断模型为什么这么分，也方便发现标注样本是否选得不好。

## 标注建议

- good/bad 各选 3 个左右通常就够第一版使用。
- 优先选“典型样本”，不要一开始就选边界样本。
- 如果某类 bad case 很多样，例如循环、答非所问、输出过短、执行路径混乱，尽量让 bad 标注覆盖不同类型。
- 跑完后优先复查 score 在 `0.4` 到 `0.6` 的样本，这些是模型最不确定的样本。

## 优点

- 比纯规则更能贴近你的实际 good/bad 标准。
- 标注量很小，适合快速迭代。
- 不依赖外部包，也不依赖网络。
- 输出带分数和最近标注样本，便于排查误判。

## 局限

- 每类标注太少时，模型只能学到粗粒度边界。
- 如果标注样本本身不典型，分类会被带偏。
- 该方法学习的是 trace 行为形态，不会像 LLM 那样真正理解业务答案质量。后续如果需要，可以在此基础上加入 LLM 对最终 result 的语义评估分。

## 使用方式

输入文件夹：

```powershell
python .\supervised\classify_supervised.py <trace文件夹> --labels labels.csv
```

输入单个 JSONL 文件：

```powershell
python .\supervised\classify_supervised.py .\data.jsonl --labels labels.csv
```

递归读取子目录：

```powershell
python .\supervised\classify_supervised.py <trace文件夹> --labels labels.csv --recursive
```

调整分类阈值：

```powershell
python .\supervised\classify_supervised.py <trace路径> --labels labels.csv --threshold 0.55
```

输出到 TSV：

```powershell
python .\supervised\classify_supervised.py <trace路径> --labels labels.csv --output result.tsv
```

指定参与评判的顶层字段：

```powershell
python .\supervised\classify_supervised.py <trace路径> --labels labels.csv --fields query,plan_list
```

输出文件使用 `utf-8-sig` 编码，便于 Windows Excel 直接打开中文。
