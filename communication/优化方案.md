# Agent Trace 分类优化方案

## 1. 优化目标

本阶段优化以提升 badcase 识别 precision 为主目标，同时监控 recall，避免为了提高 precision 导致 badcase 大量漏检。

建议将目标拆成两层：

- 主目标：提升 badcase precision，减少 goodcase 被误判为 badcase 的比例。
- 约束目标：recall 不应严重下降，需要设置最低可接受召回线。
- 辅助目标：输出可解释的 badcase 原因，但不依赖人工预先设计固定 taxonomy。

在当前已有结果中，监督特征法 precision=0.80、recall=0.52，是 precision 最好的基线；规则法默认 precision=0.77、recall=0.43，也具备较高精度。因此下一阶段应围绕“高置信判 bad”来设计，而不是单纯追求更多样本被判为 badcase。

## 2. 总体设计原则

### 2.1 采用分层判别，而不是单模型一刀切

建议将系统拆成三层：

- 第一层：高置信规则拦截。只拦截非常明确的 badcase，例如结构缺失、关键结果为空、明显连续重复、明显循环。
- 第二层：结构化特征模型。根据 trace 行为特征计算 badcase 风险分，并设置偏高阈值，保证 precision。
- 第三层：语义复核模型。只处理边界样本或需要语义判断的样本，重点审查“结果是否真正满足 query”。

最终输出不只给 `goodcase` / `badcase`，还应给出：

- `label`
- `confidence`
- `decision_stage`
- `evidence`
- `risk_factors`
- `auto_failure_cluster`
- `recall_guard_status`

### 2.2 使用双阈值机制保护 precision 和 recall

建议不要只用一个 threshold 直接二分类，而是使用三个区间：

- 高置信 badcase 区间：直接判 badcase。
- 高置信 goodcase 区间：直接判 goodcase。
- 不确定区间：进入 LLM 语义复核、用户补充信息流程或人工抽检。

这样可以把系统优化目标从“所有样本必须立即二分”调整为“只有证据足够时才给强判定”，更符合 precision 优先的目标。

### 2.3 badcase taxonomy 采用自动发现机制

不建议人工预先设计一套固定 badcase taxonomy。原因是当前任务还处于探索期，过早固定分类会限制模型发现新失败模式。

建议使用“自动归因 + 动态聚类”的方式形成 taxonomy：

- 从每条 badcase 候选中抽取风险证据，例如重复模式、空结果、短结果、query/result 低相关、缺失最终大纲等。
- 对风险证据、trace 摘要、关键失败步骤做 embedding 或文本聚类。
- 使用 LLM 或规则摘要器为每个 cluster 自动生成短名称。
- 随着新数据进入，允许 cluster 合并、拆分和重命名。

输出的 badcase 类型不是固定枚举，而是动态生成的 `failure_cluster`，例如：

- `cluster_03: 多步执行后最终产物缺失`
- `cluster_07: task 看似完成但 result 与 query 主题弱相关`
- `cluster_11: 检索或生成步骤重复但没有新增信息`

这种方式既能保留通用规则，又能让模型自主发现新模式。

## 3. 场景一：只有无标注 trace

### 3.1 场景特点

输入只有 trace，没有人工标注。此时无法直接优化真实 precision，只能通过高置信规则、异常检测和自一致验证来构建高精度 badcase 候选集。

目标不是找出所有 badcase，而是先找到“非常可能是 badcase”的样本。

### 3.2 推荐方案：高置信规则 + 异常检测 + LLM 复核

第一步，抽取统一特征。

建议把当前无监督和监督脚本中的特征抽取统一成一个 `feature_utils.py`，至少包含：

- `step_count`
- `result_nonempty_ratio`
- `final_result_length`
- `max_same_run`
- `max_loop_repeats`
- `unique_task_ratio`
- `unique_command_ratio`
- `unique_result_ratio`
- `missing_command_ratio`
- `query_task_overlap`
- `query_final_result_overlap`
- `has_outline_task`
- `final_result_has_outline_structure`

第二步，构建高 precision 规则。

无标注场景下应使用保守规则，只在证据很强时判 badcase：

- `plan_list` 缺失或不是 list。
- `plan_list` 为空。
- 所有 task 的 `result` 都为空。
- 最终 task 的 `result` 为空，且没有任何中间 task 产出大纲。
- 连续重复或循环达到较高阈值，例如重复 3 次以上。
- 多个 task 的 result 完全相同，且最终结果没有新增信息。

这些规则的目标是高 precision，不追求覆盖所有 badcase。

第三步，做无监督异常检测。

对 trace 特征向量做 robust scaling，然后使用以下方法寻找异常簇：

- Isolation Forest
- Local Outlier Factor
- One-Class SVM
- 基于中位数和 IQR 的稳健异常分数

异常检测结果不要直接作为 badcase 标签，而是作为 `risk_score`。

第四步，使用 LLM 做高置信复核。

只把满足以下条件的样本送给 LLM：

- 命中高置信规则。
- 异常分数很高。
- trace 行为正常但最终结果和 query 语义相关性低。

LLM 的任务不是自由判断，而是做证据校验：

```text
请判断以下 badcase 证据是否足以支持“该 trace 未有效完成用户请求”。
如果证据不足，请输出 goodcase_or_uncertain。
只有当证据明确时输出 badcase。
```

这样可以避免纯 LLM 过度乐观或过度主观。

### 3.3 自动 taxonomy 生成

对被高置信判为 badcase 的样本，自动生成失败模式：

1. 提取结构化证据：规则命中项、异常特征、关键 task、最终 result 摘要。
2. 生成一段失败原因摘要，例如“多次重复生成大纲但最终结果为空”。
3. 对失败原因摘要做 embedding 聚类。
4. 给每个 cluster 自动生成名称。
5. 输出 cluster 分布和代表样本。

这样可以在无标注条件下得到初始 badcase taxonomy，并为后续少量标注提供候选标签。

### 3.4 评估方式

无标注场景不能计算真实 precision/recall，建议采用替代评估：

- 抽样人工核查高置信 badcase，估计 precision。
- 统计高置信 badcase 覆盖率，作为弱 recall 参考。
- 对每个自动 cluster 抽样核查，观察是否混入大量 goodcase。
- 用不同阈值生成 precision-coverage 曲线。

重点看：

- 高置信 badcase 抽样 precision 是否足够高。
- 高风险 cluster 是否稳定。
- 是否存在明显误杀的规则。

### 3.5 适用输出

无标注场景建议输出三类结果：

- `badcase_high_confidence`
- `goodcase_or_uncertain`
- `needs_review`

不建议强行输出完整二分类，因为这会牺牲 precision。

## 4. 场景二：有少量标注 trace

### 4.1 场景特点

输入包含少量人工标注 trace。此时可以直接围绕 precision 优化模型，但样本量较少，不能依赖复杂大模型训练。

目标是用少量标注校准规则、阈值和轻量模型。

### 4.2 推荐方案：规则候选 + 轻量监督模型 + 阈值校准

第一步，保留高置信规则作为 precision 基线。

规则输出不要只给标签，还要输出命中原因：

- `missing_plan_list`
- `empty_plan_list`
- `all_results_empty`
- `final_result_empty`
- `strong_repetition`
- `strong_loop`
- `same_result_repetition`

这些原因可以作为模型特征。

第二步，训练轻量监督模型。

建议先从以下模型开始：

- Logistic Regression
- Random Forest
- LightGBM 或 XGBoost
- 校准后的线性 SVM

在样本少的情况下，优先选择可解释、稳定、参数少的模型。当前原型距离分类可以保留为 baseline，但建议新增标准监督模型。

第三步，以 precision 为主做阈值搜索。

不要默认使用 0.5 threshold。建议在验证集或交叉验证中搜索阈值：

```text
目标函数 = precision 最大化
约束条件 = recall >= recall_floor
```

其中 `recall_floor` 可以先设为当前 best precision 模型 recall 的 80%-90%。例如当前监督法 recall=0.52，则初始 recall floor 可设为 0.42-0.47，避免 recall 严重下降。

第四步，输出三段式判定。

对模型概率使用双阈值：

- `bad_prob >= high_bad_threshold`：判 badcase。
- `bad_prob <= low_bad_threshold`：判 goodcase。
- 中间区间：进入 LLM 复核或人工复核。

`high_bad_threshold` 应围绕 precision 调高，`low_bad_threshold` 用于减少无谓复核。

### 4.3 少量标注下的数据增强

可以使用以下方式提升模型稳定性：

- 对 badcase 进行近邻检索，找到无标注 trace 中相似样本，作为伪标签候选。
- 对模型高置信 badcase 做人工抽检，通过后加入训练集。
- 使用 active learning，每轮优先让用户标注模型最不确定的样本。
- 使用 cluster sampling，每个失败 cluster 只抽代表样本标注，减少重复标注成本。

伪标签必须谨慎使用：

- 只把高置信样本加入训练。
- 伪标签权重低于人工标签。
- 每次加入伪标签后都要检查 precision 是否下降。

### 4.4 自动 taxonomy 生成

有少量标注时，taxonomy 不需要人工设计，但可以由标注反馈校准。

流程建议：

1. 使用无标注方案生成自动 failure cluster。
2. 将少量人工标注样本映射到 cluster。
3. 统计每个 cluster 的 badcase purity。
4. 高 purity cluster 可作为高置信 badcase 规则候选。
5. 低 purity cluster 进入复核或拆分。

这样 taxonomy 仍然是自动发现的，但能被少量标注验证质量。

### 4.5 评估方式

少量标注场景建议使用：

- Stratified K-Fold 交叉验证。
- 按 source 分组评估，避免某个来源过拟合。
- precision-recall curve。
- high-confidence badcase precision。
- recall floor 监控。
- 误杀 goodcase 样本分析。

核心指标建议：

```text
badcase_precision
badcase_recall
badcase_f1
badcase_high_confidence_precision
review_rate
recall_drop_ratio
```

其中 `review_rate` 表示被送入不确定区间的比例。precision 优先场景下，适当提高 review_rate 是可以接受的。

## 5. 场景三：可以和用户互动补充信息

### 5.1 场景特点

用户可以补充信息，例如 Agent 的任务描述、期望输出标准、工具说明、trace 字段含义、已知失败案例。这是最有潜力提升 precision 的场景，因为很多误判来自“系统不知道什么才算真正完成任务”。

目标是用最少的问题换取最大的信息增益。

### 5.2 推荐方案：交互式任务画像 + 自适应评审器

第一步，向用户收集任务画像。

建议优先询问以下信息：

- Agent 的目标任务是什么。
- goodcase 的最低验收标准是什么。
- 最终输出应出现在哪个字段或哪个 task 中。
- 哪些结果即使非空也应视为失败。
- 是否存在必须调用或禁止调用的工具。
- 是否有代表性 goodcase 和 badcase 样例。

这些信息不需要用户一次性全部提供。系统可以根据当前误判样本动态追问。

第二步，把用户信息转成可执行评审规则。

例如用户说明“最终必须生成 PPT 大纲，且至少包含标题、章节和每章要点”，则可以转成：

- 最终 result 必须非空。
- 最终 result 应包含多级结构。
- 最终 result 应覆盖 query 主题。
- 只出现泛化建议而无大纲结构时，不应判 goodcase。

这些规则可以作为 LLM prompt 的约束，也可以变成结构化特征。

第三步，构建交互式误判修正流程。

每轮系统选择最有价值的样本给用户：

- 高置信 badcase 但模型证据较少的样本。
- 模型和规则冲突的样本。
- 自动 cluster 中代表性最强的样本。
- 误杀风险最高的 goodcase 候选。

用户只需要回答简单问题：

- 这个 trace 是否实际完成任务？
- 如果失败，最关键失败点是什么？
- 这个失败是否和某个已知模式相同？
- 这类样本以后应如何判？

第四步，将用户反馈沉淀为三类知识。

- 验收标准：用于 LLM prompt 和规则。
- 标签样本：用于监督模型训练。
- cluster 解释：用于自动 taxonomy 命名和合并。

### 5.3 用户信息如何提升 precision

precision 低通常意味着误杀 goodcase。用户互动能重点解决以下问题：

- 明确“什么程度的输出算可接受”，减少对短但有效结果的误杀。
- 明确“哪些中间失败不影响最终完成”，减少规则误判。
- 明确“哪些工具调用路径是正常流程”，减少异常检测误判。
- 提供少量典型 goodcase，帮助模型学习正常行为边界。

因此，在可互动场景下，不应只让用户补 badcase，也要主动收集 goodcase 标准和代表样例。

### 5.4 自动 taxonomy 生成

用户互动场景下，taxonomy 仍由系统自动生成，但用户可以做轻量确认。

推荐流程：

1. 系统自动聚类失败原因。
2. 系统为 cluster 生成名称和解释。
3. 用户只确认“这个名称是否合理”“是否应和另一个 cluster 合并”。
4. 系统用用户反馈更新 cluster 命名和边界。

用户不需要从零设计 taxonomy，只需要做低成本校准。

## 6. 推荐的综合系统架构

### 6.1 离线训练与分析流程

建议新增一个统一 pipeline：

```text
trace 输入
 -> trace schema 校验
 -> 特征抽取
 -> 高置信规则判断
 -> 无监督风险评分
 -> 监督模型评分
 -> LLM 语义复核
 -> 双阈值决策
 -> 自动 failure cluster
 -> 指标报告与样本报告
```

### 6.2 在线或批量预测流程

预测时建议按以下顺序：

1. Schema 校验失败，直接进入 badcase 候选。
2. 高置信规则命中，进入 badcase 候选。
3. 监督模型 bad_prob 极高，判 badcase。
4. 监督模型 bad_prob 极低，判 goodcase。
5. 中间样本进入 LLM 语义复核。
6. LLM 仍不确定时输出 `needs_review`。

在 precision 优先场景中，`needs_review` 比错误判 badcase 更可接受。

### 6.3 LLM 复核输入格式

LLM 不应只看原始 trace，而应看到结构化摘要：

```json
{
  "query": "...",
  "task_goal": "...",
  "trace_summary": {
    "step_count": 8,
    "final_result_length": 23,
    "has_outline_task": true,
    "max_loop_repeats": 1,
    "result_nonempty_ratio": 0.75
  },
  "risk_evidence": [
    "最终结果长度较短",
    "query 与最终 result 关键词重合低"
  ],
  "candidate_label": "badcase",
  "review_instruction": "只有当证据足以说明任务未完成时，才确认 badcase。"
}
```

这样能让 LLM 主要做证据判断，而不是重新猜标签。

## 7. 阈值与指标设计

### 7.1 阈值搜索策略

建议使用以下策略选择阈值：

```text
在验证集上枚举 high_bad_threshold
筛选 recall >= recall_floor 的阈值
在满足 recall_floor 的候选中选择 precision 最高者
如果 precision 接近，选择 review_rate 更低者
```

当标注很少时，阈值不要频繁微调，应优先使用稳定区间。

### 7.2 recall 下降监控

建议定义：

```text
recall_drop_ratio = (baseline_recall - current_recall) / baseline_recall
```

如果 `recall_drop_ratio > 20%`，说明 precision 提升可能代价过高，需要回调阈值或增加 LLM 复核覆盖。

### 7.3 建议指标面板

每次实验输出：

- badcase precision
- badcase recall
- badcase F1
- high-confidence badcase precision
- high-confidence badcase coverage
- needs_review rate
- recall_drop_ratio
- 按 source 的 precision/recall
- 按自动 failure cluster 的 precision/recall
- top false positives
- top false negatives

其中 `top false positives` 对 precision 优化最重要，因为它直接展示 goodcase 被误杀的原因。

## 8. 具体落地步骤

### 阶段一：先做可复现的 precision 优化基线

建议优先完成：

- 抽取统一 `feature_utils.py`。
- 改造规则法，输出 `risk_factors`。
- 在少量标注或现有 train/test 上训练标准监督模型。
- 增加 threshold search。
- 增加 recall floor。
- 输出 false positive 样本分析。

阶段目标：

- 得到一个 precision 高于当前 0.80，且 recall 不低于设定下限的结构化基线。

### 阶段二：加入自动 taxonomy 与聚类分析

建议完成：

- 为每个 badcase 候选生成失败摘要。
- 对失败摘要和结构化风险特征做聚类。
- 自动生成 cluster 名称。
- 输出每个 cluster 的样本数、代表样本、平均风险分、人工标注纯度。

阶段目标：

- 不依赖人工预设 taxonomy，也能看到 badcase 类型分布。

### 阶段三：加入 LLM 语义复核

建议完成：

- 只对边界样本调用 LLM。
- LLM 输入使用 trace 摘要 + 风险证据 + 用户任务描述。
- LLM 输出必须包含是否确认 badcase、证据是否充分、置信度。
- 对 LLM 输出做 JSON 解析和失败兜底。

阶段目标：

- 降低 false positive，提高 precision，同时尽量追回一部分语义型 badcase。

### 阶段四：加入用户互动闭环

建议完成：

- 自动挑选最值得用户标注的样本。
- 支持用户补充任务描述和验收标准。
- 将用户反馈转成规则、prompt 约束和训练样本。
- 每轮反馈后重新评估 precision 和 recall。

阶段目标：

- 让系统能随着用户知识注入持续变准。

## 9. 三种场景的推荐方案对比

| 场景          | 主方案                   | precision 提升手段       | recall 保护手段              | taxonomy 方式       |
| ----------- | --------------------- | -------------------- | ------------------------ | ----------------- |
| 只有无标注 trace | 高置信规则 + 异常检测 + LLM 复核 | 只输出强证据 badcase       | 输出 `needs_review`，不强行二分类 | 风险证据聚类 + 自动命名     |
| 有少量标注 trace | 规则特征 + 轻量监督模型 + 阈值校准  | 以 precision 为目标搜索高阈值 | 设置 recall floor 和交叉验证    | cluster purity 校准 |
| 可和用户互动      | 任务画像 + 自适应评审器 + 主动学习  | 用验收标准减少误杀            | 让用户补充边界样本和典型 badcase     | 系统自动生成，用户轻量确认     |

## 10. 最终建议

下一阶段建议选择“少量标注 trace 场景”作为主线，因为它最适合实现 precision 优化闭环；同时保留“无标注 trace 场景”作为冷启动方案，并把“用户互动场景”作为增强模块逐步接入。

推荐优先级如下：

1. 先把现有特征抽取统一，建立可复现的监督模型和阈值搜索。
2. 以 precision 最大化为目标，同时设置 recall floor。
3. 引入双阈值和 `needs_review`，避免低证据样本被强判 badcase。
4. 使用自动聚类生成 badcase taxonomy，不人工预设固定分类。
5. 对边界样本加入 LLM 证据复核。
6. 通过用户提供任务描述、验收标准和少量样例继续提升 precision。

最终系统应从“单脚本二分类”升级为“高置信 badcase 检出 + 不确定样本复核 + 自动失败模式发现”的组合系统。这样最符合当前 precision 优先、recall 不能严重下降、taxonomy 不靠人工设计的优化目标。
