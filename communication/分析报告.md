# Agent Trace 分类项目分析报告

## 1. 项目目标与当前定位

本项目的核心任务是对 Agent trace 进行质量判别，将样本快速划分为 `goodcase` 或 `badcase`，并为后续 badcase 原因分析提供入口。结合现有代码与“项目现状描述.md”，可以判断该仓库当前处于“方法探索型原型阶段”，而不是“可稳定复现、可持续迭代的工程化系统”阶段。

当前仓库已经围绕同一任务实现了多条技术路线：

- 基于规则的快速判别：`classify_traces.py`
- 基于通用 LLM 的直接分类：`classify_llm.py`
- 基于 Agent 审查式提示词的分类：`classify_agent.py`
- 规则 + LLM 的混合分类：`classify_hybrid.py`
- 基于行为特征的无监督分类：`unsupervised/classify_unsupervised.py`
- 基于 train/test 划分的原型式监督分类：`supervised/classify_supervised.py`
- 基于 APO 的提示词优化实验：`supervised/classify_APO.py`

从方向覆盖上看，项目已经探索了“规则、特征、监督学习、LLM、混合、提示词优化”六类路径，说明问题理解比较充分，探索广度是这个项目的明显优点。

## 2. 当前代码结构与能力概览

项目结构比较清晰，核心由“分类脚本 + 统一评估工具”组成：

- `trace_eval_utils.py`：负责 metadata 读取、trace 加载、指标统计、结果输出，是全项目公共底座。
- 顶层 `classify_*.py`：偏向规则和 LLM 路线。
- `unsupervised/`：偏向人工设计特征 + 打分公式。
- `supervised/`：偏向 train/test 驱动的特征原型分类和 APO 提示词优化。

现有公共评估能力已经具备以下优点：

- 标签、split、source 的数据结构统一。
- badcase 作为正类的 precision / recall / F1 计算口径统一。
- 支持按 source 做切片分析。
- 支持输出 markdown 指标文件和逐样本 TSV。

这说明项目已经具备“比较不同方案效果”的最小实验平台能力。

## 3. 各方案效果与适用性分析

结合说明文档中的实验结果，可以得到以下判断。

### 3.1 规则法

规则法当前结果：

- 默认：precision=0.77, recall=0.43
- `repeat-threshold=2`：precision=0.70, recall=0.61

优点：

- 可解释性强，运行成本低，适合作为第一层快速过滤。
- 对“重复执行、循环、无有效结果”这类显性 badcase 较敏感。

问题：

- 规则较硬，主要依赖 `plan_list` 是否存在、是否重复、是否出现“生成大纲”且结果非空。
- 对语义层面的 badcase 几乎无感，例如“答非所问”“结果虽非空但无效”“大纲质量极低”。
- 对 goodcase 的判定证据偏弱，本质更像“未发现明显错误即偏向 good”。

结论：

规则法适合保留为基线和第一层拦截器，但不适合作为最终主模型。

### 3.2 直接 LLM 分类

`classify_llm.py` 的结果是把所有样本都判成 goodcase，precision=0，recall=0。

这说明至少存在以下一种或多种问题：

- 提示词约束不够强，模型默认偏乐观。
- 输入 trace 太长或太噪，模型没有抓住 badcase 信号。
- 输出解析虽然做了兼容，但模型判别逻辑本身没有真正建立起来。
- 任务定义对模型来说不够“可执行”，缺少明确负例标准和强制审查步骤。

结论：

直接让 LLM 一步到位判标签，在这个任务上效果很差，不建议继续单独投入。

### 3.3 Agent 审查式 / 混合法

混合法结果为：

- precision=0.59, recall=0.69

这比纯 LLM 明显更可用，说明“规则先拦截显性 badcase，再让 LLM 处理复杂边界样本”的方向是对的。它的价值不在于当前结果已经最好，而在于架构上更符合任务特点：

- 显式异常交给规则，低成本且稳定。
- 隐式异常交给 LLM，补规则盲区。

但现阶段它仍有明显不足：

- LLM 调用依赖外部 `UniAIGC`，项目内没有可复现的运行闭环。
- prompt 虽然比直接分类更严格，但没有形成稳定的 few-shot 或错误驱动优化机制。
- 规则与 LLM 的职责边界还比较粗，仍然只是串联，不是深度融合。

结论：

混合路线是最值得继续投入的主线之一。

### 3.4 无监督特征法

无监督方案在 `threshold=0.75` 时：

- precision=0.49, recall=0.74

它最大的优点是 badcase 召回高，说明手工设计的行为特征确实能捕捉到不少异常模式。当前实现中已经抽取了较丰富的行为信号，例如：

- 步数
- task / command / result 多样性
- 连续重复
- 循环模式
- result 非空比例
- 最终结果长度
- query 与 task 的重合度
- entropy 类特征

优点：

- 不依赖标注训练，可快速试错。
- 特征可解释，便于沉淀 badcase 模式。
- 对后续监督模型与规则升级都有启发作用。

问题：

- 目前是手工打分公式，参数和权重依赖经验调节。
- 精度偏低，误杀 goodcase 较多。
- 特征虽丰富，但尚未形成“失败原因标签体系”，仍停留在分类而非诊断。

结论：

无监督方案非常适合继续作为“特征工程与错误模式挖掘平台”，但不建议单独作为最终线上判别器。

### 3.5 有监督特征法

监督方案在 `threshold=0.85` 时：

- precision=0.80, recall=0.52

这是当前说明文档中 precision 最好的方案，说明基于 train/test 的特征原型法已经具备一定区分能力。相比无监督法，它更稳、更有收敛方向。

优点：

- 使用训练集统计特征中位数和尺度，能把经验特征转化为更系统的分类过程。
- nearest good / nearest bad 的输出有助于解释。
- 对中小规模数据较友好，训练成本低。

问题：

- 模型本质仍然较简单，更像“原型距离分类”而非真正的判别模型。
- 特征和无监督脚本大量重复，存在维护成本。
- 当前召回偏低，说明 badcase 识别仍不够激进。

结论：

监督特征法是当前最稳的传统机器学习方向，建议作为“结构化主基线”继续强化。

### 3.6 APO 提示词优化

APO 当前结果在 `batch=2, epoch=3` 时 F1=0.42。

这说明“用提示词优化提升分类能力”的思路尚未跑通，原因很可能包括：

- 基础分类 prompt 本身不稳定，优化对象不够强。
- teacher 与 student 使用同一模型，优化增益有限。
- 用训练小 batch 生成自然语言梯度，再在 test 上频繁回看，容易导致实验逻辑混杂。
- 当前优化目标只盯 F1，但数据规模可能不足以支撑稳定迭代。

结论：

APO 暂时更像研究实验，不建议作为近期主线。

## 4. 项目当前的主要优势

### 4.1 方法探索广，已经形成有效对比面

项目不是只押一个方法，而是已经把规则、特征、监督、LLM、混合都跑过一遍。这使得下一步优化有充分参照，不容易盲目投入。

### 4.2 评估口径基本统一

`trace_eval_utils.py` 统一了数据加载、指标输出、按 source 统计，这一点很关键。说明后续继续迭代时，至少有一致的比较基准。

### 4.3 特征理解较深入

无监督和监督方案中已经沉淀出一批与 trace 行为强相关的特征，这些特征后续无论用于规则升级、监督建模还是给 LLM 提供结构化上下文，都很有价值。

### 4.4 任务目标明确，业务边界清晰

当前任务聚焦于“PPT 大纲生成 Agent 的 trace 质检”，目标场景相对集中，有利于尽快做出一版实用方案。

## 5. 当前存在的核心问题

### 5.1 工程化程度偏低，复现成本高

从代码看，多个脚本仍保留硬编码的 `INLINE_ARGS` 路径，依赖本地数据路径；LLM 路线依赖 `UniAIGC`，但仓库内没有完整依赖说明、配置说明和运行约束。意味着：

- 新同学接手后很难直接跑通。
- 实验结果难以复现。
- 很难接入自动化评测或持续迭代流程。

### 5.2 代码重复较多，方法之间没有真正抽象复用

例如 LLM/Agent/Hybrid/APO 中都重复实现了：

- JSON 提取逻辑
- 标签归一化逻辑
- 批处理逻辑
- prompt 组装和结果解析逻辑

监督/无监督方案中也有大量重复的特征抽取思路。这会带来两个问题：

- 一处修复，多个文件要同步改。
- 方法看似很多，但真实可维护性不高。

### 5.3 当前更像“分类实验”，还不是“badcase 诊断系统”

项目目标里提到“对于 badcase，进一步分析失败原因”，但当前主产物仍以二分类为主。虽然 Agent/Hybrid 有 `badcase_type` 字段，但并没有形成稳定、闭环的错误类型体系。

也就是说，现在能“分坏好”，但还不能稳定回答“为什么坏、坏在哪一步、应如何修复”。

### 5.4 数据与特征仍偏弱语义化

现有特征更偏行为统计特征，对以下问题感知有限：

- 结果内容是否真正满足 query
- 大纲结构是否完整
- 输出是否只是空泛模板
- task 结果之间是否逻辑连贯

这会导致模型对“语义低质 badcase”识别不足。

### 5.5 实验结论还没有沉淀成明确主线

目前每条路线都做了一些尝试，但缺少一个清晰的结论：

- 哪条路线是主路线？
- 哪条路线只是基线或参考？
- 下一阶段要优化的是 precision 还是 recall？
- 业务上更不能接受漏判 badcase，还是误杀 goodcase？

如果这些没有先定清楚，后续优化会继续分散。

### 5.6 仓库卫生和交付形态仍偏实验化

当前仓库中没有看到：

- 完整 README
- 依赖清单
- 样例数据说明
- 自动化测试
- 统一实验命令入口

同时还保留了 `__pycache__` 文件，说明仓库清理和交付规范还没有建立起来。

## 6. 对现有技术路线的总体判断

如果从“近期做出可用版本”角度看，建议这样判断各路线角色：

- 规则法：保留，作为第一层高精度拦截器和对照基线。
- 无监督特征法：保留，作为特征探索与召回增强参考。
- 有监督特征法：重点投入，作为当前最稳的结构化主模型。
- 混合法：重点投入，作为最有潜力的业务主方案。
- 纯 LLM 法：降级为对照实验，不建议单独推进。
- APO：降级为研究储备，不建议近期继续主投。

换句话说，项目下一步不应再平均用力，而应收敛为“双主线”：

- 主线 A：规则 + 特征监督模型
- 主线 B：规则 + LLM 语义复核

## 7. 下一步优化方向建议

### 7.1 第一优先级：先完成工程收敛与实验闭环

这是最该优先做的，不然任何算法优化都会反复失真。

建议动作：

- 补齐 `README`，说明任务、数据格式、运行方式、指标口径。
- 增加 `requirements.txt` 或等价依赖说明。
- 去掉脚本中的硬编码 `INLINE_ARGS`，改为纯命令行运行。
- 将公共逻辑抽到统一模块，例如 `llm_utils.py`、`feature_utils.py`、`prompt_utils.py`。
- 清理 `__pycache__` 等无关文件。
- 固化一组标准实验命令，确保不同方案可一键复现。

预期收益：

- 让项目从“个人实验脚本集合”升级为“团队可协作实验仓库”。

### 7.2 第二优先级：建立 badcase 类型体系，而不只做二分类

当前任务真正的价值不只是“筛坏样本”，而是帮助定位 Agent 失败模式。建议尽快定义一个稳定的 badcase taxonomy，例如：

- 循环/重复执行
- 缺失关键结果
- 结果为空或过短
- 结果与 query 无关
- 任务链不完整
- 最终产物质量低
- 指令理解偏移

然后把现有方案都往这个体系上对齐：

- 规则法输出明确的规则命中类型
- 特征法输出风险因子
- LLM/Hybrid 输出标准化 `badcase_type`

预期收益：

- 后续不仅能分类，还能做失败原因分布分析与优化闭环。

### 7.3 第三优先级：把监督特征法升级成真正的结构化模型

当前监督法已经是一个不错的起点，但模型仍然偏简单。建议逐步升级：

- 先统一特征抽取代码，沉淀成单独模块。
- 在同一套特征上尝试更标准的模型，如逻辑回归、随机森林、XGBoost 或轻量 GBDT。
- 通过交叉验证而不是单点 threshold 手调，稳定评估 precision/recall/F1。
- 输出特征重要性，帮助理解 badcase 关键模式。

预期收益：

- 以较低成本把当前 best precision 路线做实。

### 7.4 第四优先级：把混合法做成真正两阶段系统

当前混合法是最有业务潜力的方向，建议从“串联脚本”升级为“两阶段判别系统”：

第一阶段：

- 用规则和轻量特征模型拦截显性 badcase。
- 对高置信 goodcase 直接放行。

第二阶段：

- 只把边界样本交给 LLM 做语义复核。
- 给 LLM 的输入不只包含原始 trace，还包含结构化摘要和风险提示，例如：
  - 重复次数
  - 最终结果长度
  - query-task overlap
  - 是否有最终大纲任务
  - 规则/特征模型的初判原因

预期收益：

- 降低 LLM 成本。
- 提升复杂 badcase 的识别能力。
- 增强解释性。

### 7.5 第五优先级：从“特征统计”升级到“语义质量判断”

这个项目的难点不只是流程异常，更是内容质量异常。建议补充语义级特征或语义判别器，例如：

- 最终大纲是否包含多级结构
- 标题是否覆盖 query 中的主题关键词
- 各章节是否存在明显模板化空话
- plan_list 中关键 result 是否前后呼应
- 最终结果是否足够长且信息密度达标

这部分可以有两种做法：

- 结构化规则/特征方式：先做低成本语义 proxy
- LLM 审核方式：只在边界样本上做深审查

### 7.6 第六优先级：重新定义指标目标与业务取舍

下一步优化前，建议先明确业务目标：

- 如果目标是“尽量别放过 badcase”，就优先 recall。
- 如果目标是“不要误伤太多正常 trace”，就优先 precision。
- 如果后面还有人工复核，系统可以偏 recall。
- 如果要直接自动拦截，系统需要偏 precision。

基于当前结果，我更建议把系统设计成“双阈值”：

- 高置信 badcase：直接拦截
- 高置信 goodcase：直接通过
- 中间区间：进入 LLM/人工复核

这样比单阈值二分更贴合实际使用场景。

## 8. 推荐的阶段性路线图

### 阶段一：一周内可完成

- 完成仓库整理、依赖说明、统一运行入口。
- 固化标准评测集和基线结果。
- 清理重复代码，抽公共模块。
- 定义 badcase 类型标签体系。

阶段目标：

- 让所有方案稳定可跑、结果可复现、输出可比。

### 阶段二：两到三周建议完成

- 把无监督和监督特征抽取统一。
- 在同一特征集上训练更标准的监督模型。
- 输出 confusion matrix + source 分析 + badcase_type 分析。
- 找出 top 错误样本，做人审回放。

阶段目标：

- 得到一个稳定、可解释、可复现的结构化主基线。

### 阶段三：随后重点推进

- 构建两阶段混合系统。
- 只让 LLM 处理边界样本和语义低质样本。
- 把规则命中信息、特征摘要一起送给 LLM。
- 形成“分类 + 原因诊断”的联合输出。

阶段目标：

- 做出最接近业务可用形态的方案。

## 9. 最终建议结论

综合来看，这个项目的方向是对的，探索面也足够广，最大的短板不在“没有思路”，而在“还没有从实验堆叠收敛为一条可持续优化的主线”。

最建议的下一步不是继续平铺更多新方法，而是按以下顺序收敛：

1. 先补工程化和复现闭环。
2. 再建立 badcase 类型体系。
3. 以监督特征模型作为稳定主基线。
4. 以规则 + LLM 复核作为业务主方案重点推进。
5. 暂缓纯 LLM 和 APO 的继续主投。

如果执行到位，项目下一阶段最有希望形成的成果不是“某个单点 F1 再涨一点”，而是一个真正能支撑业务使用的 trace 质量判别与失败诊断系统。
