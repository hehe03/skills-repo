# 无标注 Trace 场景优化方案

## 1. 场景目标

本方案面向只有无标注 trace 的情况，即没有人工标注的 `goodcase` / `badcase`，只能依靠 Agent 任务流程、通用质量规则、trace 行为特征和模型自检来发现高置信 badcase。

本阶段目标如下：

- 以提升 badcase precision 为主，不轻易把样本判为 badcase。
- 对 recall 做间接保护，避免只发现极少数显性错误。
- 不人工预设固定 badcase taxonomy，而是通过规则证据、流程偏离和聚类自动发现失败模式。
- 结合“高层客户交流助手”的业务流程，设计可解释、可执行的无标注检测方法。

在无标注场景中，不建议强制输出完整二分类。更合适的输出是：

- `badcase_high_confidence`：证据非常充分，可高置信判为 badcase。
- `goodcase_likely`：流程和结果均未发现明显异常，可暂认为 goodcase。
- `needs_review`：证据不足，需要 LLM 复核、人工抽查或用户补充信息。

这样可以最大限度保护 precision。

## 2. Agent 任务流程理解

该 Agent 的角色是面向高层客户的交流助手，核心工作不是简单生成文本，而是按业务流程逐步帮助用户形成交流大纲。

标准流程分为三大部分。

第一部分：要素澄清和检查。

- 用户输入可以是一句话，也可以是策划报告文件。
- Agent 需要向用户澄清关键要素。
- Agent 需要检查要素是否完备。
- 如果关键要素缺失，应先澄清，而不是直接进入后续任务。

第二部分：要素推荐。

- 只有在用户同意的情况下，Agent 才帮助用户收集交流要素。
- 要素推荐包含多个按顺序执行的任务：
- 历史交流话题补充与确认。
- 客户痛点话题补充与确认。
- 相似客户案例推荐与确认。
- 客户合作信息补充与确认。

说明：用户补充信息中提到“有6个任务”，但列出的任务为4类。无标注方案不应据此强行判错，而应把“实际任务清单与期望任务数不一致”记录为 `needs_review` 风险点，后续可通过用户补充配置修正。

第三部分：大纲生成。

- Agent 根据要素报告生成交流大纲。
- 大纲应服务于高层客户交流场景，而不是普通泛化大纲。
- 如果前置要素不足，大纲生成应体现约束或向用户确认，而不是假装已完成。

## 3. 例外流程理解

该 Agent 允许从不同阶段直接开始，因此不能简单要求所有 trace 都必须完整走完三大部分。

例外一：用户直接要求要素推荐。

- 可以跳过要素澄清和检查。
- 应从要素推荐部分开始。
- 仍然需要按要素推荐任务顺序执行。
- 仍然需要在关键节点获得用户确认或展示结果。

例外二：用户直接要求生成大纲。

- 可以跳过要素澄清和要素推荐。
- 应直接进入大纲生成。
- 生成的大纲仍需与用户输入或报告内容相关。

例外三：工具执行未达目标。

- 禁止继续执行新的工具。
- 应向用户展示当前工具执行结果。
- 应询问用户是否进入下一任务。

例外四：禁止自行重复调用工具。

- 如果没有用户明确要求或新的上下文变化，Agent 不应重复调用同一个工具。
- 重复调用工具是高置信 badcase 信号，尤其是连续重复调用且结果没有新增信息时。

这些例外规则非常关键。无标注检测时，不能只看“是否完整执行标准流程”，而要先判断用户意图属于哪一种入口模式。

## 4. 无标注检测的总体架构

建议采用四层检测：

```text
trace 输入
 -> schema 与字段完整性检查
 -> 用户意图入口识别
 -> 流程一致性检查
 -> 工具调用合规性检查
 -> 结果质量与语义一致性检查
 -> 高置信 badcase 判定或 needs_review
 -> 自动失败模式聚类
```

每一层都输出证据，但只有当证据足够强时才判 `badcase_high_confidence`。

## 5. Trace 预处理与结构化摘要

对每条 trace 先抽取结构化摘要，作为后续规则和模型输入。

建议字段如下：

- `query`：用户原始输入。
- `plan_list_exists`：是否存在 `plan_list`。
- `step_count`：任务步数。
- `task_names`：任务名称序列。
- `command_names`：工具名称序列。
- `command_args`：工具参数摘要。
- `results`：每步结果摘要。
- `final_result`：最终结果。
- `empty_result_steps`：结果为空的步骤。
- `failed_tool_steps`：工具失败步骤。
- `user_confirmation_steps`：向用户确认的步骤。
- `outline_steps`：大纲生成相关步骤。
- `recommendation_steps`：要素推荐相关步骤。
- `repeat_tool_runs`：重复工具调用统计。
- `loop_patterns`：循环调用模式统计。

同时生成一段自然语言摘要：

```text
用户输入意图：...
识别入口模式：...
执行阶段：...
关键工具调用：...
最终输出：...
主要风险证据：...
```

这段摘要可用于 LLM 复核和失败模式聚类。

## 6. 用户意图入口识别

无标注场景下，首先要判断用户 query 期望 Agent 从哪个阶段开始。

建议将入口模式分为：

- `clarification_entry`：用户只给一句话或报告，需要先澄清和检查要素。
- `recommendation_entry`：用户明确要求要素推荐、补充客户信息、推荐案例、补充话题等。
- `outline_entry`：用户明确要求直接生成交流大纲。
- `unknown_entry`：无法稳定判断入口。

入口识别可以使用轻量规则 + LLM 辅助。

规则信号示例：

- query 包含“直接生成大纲”“帮我写大纲”“输出交流大纲”，倾向 `outline_entry`。
- query 包含“推荐要素”“补充话题”“客户痛点”“相似案例”“合作信息”，倾向 `recommendation_entry`。
- query 只有简短背景、客户名、文件 ID 或模糊一句话，倾向 `clarification_entry`。

如果入口不确定，不应直接判 badcase，而应标记为 `needs_review`。

## 7. 高置信 badcase 规则

以下规则以 precision 为优先，只有命中强证据时才输出 `badcase_high_confidence`。

### 7.1 基础结构异常

可直接判高置信 badcase 的情况：

- `plan_list` 缺失或不是 list。
- `plan_list` 为空，且 final result 也不存在。
- 所有步骤的 `result` 均为空。
- trace 无 query，且无法从其它字段恢复用户目标。

这些属于 trace 本身不可用，误杀风险较低。

### 7.2 工具失败后继续执行新工具

这是本业务补充信息中的强规则。

如果某一步工具执行结果显示未达目标，例如：

- result 为空。
- result 包含失败、异常、未找到、无权限、超时、无法获取、无结果等失败信号。
- 工具返回内容明确表示未完成目标。

则后续行为应是：

- 向用户展示执行结果。
- 询问是否进入下一任务。

如果 trace 显示 Agent 在未展示和未询问的情况下继续调用新工具，可判 `badcase_high_confidence`。

判定证据格式：

```json
{
  "rule": "tool_failure_continue_without_user_confirmation",
  "failed_step": 3,
  "next_tool_step": 4,
  "evidence": "第3步工具未达目标，但第4步直接调用新工具，未向用户确认"
}
```

### 7.3 自行重复调用工具

如果同一工具在相邻步骤重复调用，且满足以下条件，可判高置信 badcase：

- command name 相同。
- command args 相同或高度相似。
- 中间没有用户新输入或确认。
- 上一次 result 没有要求重试。
- 新 result 没有明显新增信息。

如果重复达到 2 次，可标记强风险；重复达到 3 次及以上，可直接判 `badcase_high_confidence`。

判定证据格式：

```json
{
  "rule": "self_repeated_tool_call",
  "steps": [5, 6, 7],
  "evidence": "连续3次调用同一工具，参数相同，且没有用户确认或新增上下文"
}
```

### 7.4 标准流程严重跳步

需要先识别入口模式，再判断是否跳步。

当入口为 `clarification_entry` 时，高置信 badcase 信号：

- 用户只提供模糊一句话或文件 ID，但 Agent 未澄清关键要素，直接生成大纲。
- Agent 未检查要素完备性，也未说明缺失项。
- 最终大纲明显建立在未确认信息上。

当入口为 `recommendation_entry` 时，高置信 badcase 信号：

- 用户要求要素推荐，但 Agent 直接生成大纲。
- Agent 没有执行任何要素推荐相关任务，也没有解释跳过原因。

当入口为 `outline_entry` 时，不应因为跳过澄清或推荐而判 badcase。只检查大纲是否生成、是否相关、是否可用。

### 7.5 要素推荐顺序严重错误

当入口为 `clarification_entry` 且用户同意进入要素推荐，或入口为 `recommendation_entry` 时，应检查要素推荐顺序。

建议先使用柔性规则：

- 若任务缺失但 Agent 给出合理说明，进入 `needs_review`。
- 若任务顺序轻微变化但结果完整，进入 `needs_review`。
- 若跳过大部分推荐任务且直接生成大纲，判高风险。
- 若工具失败后仍继续后续推荐任务，按 7.2 判高置信 badcase。

由于用户补充信息中任务数量和列举项不完全一致，当前不建议仅凭“任务数不是6个”判 badcase。

## 8. 结果质量检查

无标注场景下，结果质量检查应偏保守。

### 8.1 大纲生成质量

当入口为 `outline_entry`，或 trace 已进入大纲生成阶段，应检查最终输出。

强 badcase 信号：

- 没有最终大纲。
- 最终结果为空。
- 最终结果只是工具失败信息。
- 最终结果与用户 query 几乎无关。
- 最终结果只包含泛泛建议，没有交流大纲结构。

中等风险信号：

- 大纲过短。
- 只有一级标题，没有展开要点。
- 没有体现客户痛点、历史交流、相似案例或合作信息。
- 没有面向高层客户交流的语气和结构。

中等风险不直接判 badcase，可进入 LLM 复核。

### 8.2 要素推荐结果质量

要素推荐阶段应检查每类任务是否有可用输出和确认动作。

强 badcase 信号：

- 推荐任务 result 为空。
- 工具返回失败，但 Agent 伪装成完成。
- 没有向用户确认就把失败结果当成有效要素继续使用。

中等风险信号：

- 推荐内容非常泛化。
- 推荐内容与客户或场景无关。
- 多个推荐任务结果高度重复。

中等风险进入 `needs_review` 或 LLM 复核。

### 8.3 要素澄清质量

要素澄清阶段应检查是否围绕关键要素提问或说明缺失。

强 badcase 信号：

- 用户输入明显缺失客户、场景、目标等关键信息，但 Agent 直接生成确定性大纲。
- Agent 声称已完成要素检查，但没有任何检查结果。

中等风险信号：

- 澄清问题过泛。
- 只重复用户输入，没有提出可操作问题。

## 9. 无监督风险评分

除高置信规则外，建议构建无监督风险分，用于排序和复核，不直接作为最终 badcase 标签。

风险分可由以下部分组成：

- `schema_risk`：结构缺失风险。
- `flow_risk`：流程跳步风险。
- `tool_risk`：工具失败后继续执行、重复调用风险。
- `result_risk`：结果为空、过短、重复风险。
- `semantic_risk`：query 与最终 result 相关性低。
- `confirmation_risk`：应确认但未确认风险。

示例公式：

```text
risk_score =
  0.25 * tool_risk
  + 0.20 * flow_risk
  + 0.20 * result_risk
  + 0.15 * semantic_risk
  + 0.10 * confirmation_risk
  + 0.10 * schema_risk
```

高 precision 策略：

- `risk_score >= 0.85` 且命中至少一个强规则，判 `badcase_high_confidence`。
- `risk_score >= 0.70` 但未命中强规则，进入 `needs_review`。
- `risk_score < 0.70` 且无强规则，暂不判 badcase。

## 10. LLM 复核设计

LLM 只处理边界样本，目标是减少误杀。

### 10.1 复核输入

LLM 输入不应只包含原始 trace，应包含结构化摘要：

```json
{
  "agent_role": "面向高层客户交流助手",
  "entry_mode": "clarification_entry / recommendation_entry / outline_entry / unknown_entry",
  "query": "...",
  "expected_flow": "...",
  "actual_flow": ["...", "..."],
  "tool_events": [
    {
      "step": 3,
      "tool": "...",
      "status": "failed_or_empty",
      "next_action": "called_new_tool"
    }
  ],
  "final_result_summary": "...",
  "risk_evidence": ["...", "..."],
  "review_instruction": "只有证据足以证明未完成任务或违反硬约束时，才确认 badcase；证据不足输出 needs_review。"
}
```

### 10.2 复核输出

LLM 必须输出 JSON：

```json
{
  "review_label": "confirm_badcase / likely_goodcase / needs_review",
  "confidence": 0.0,
  "sufficient_evidence": true,
  "evidence": ["..."],
  "failure_summary": "..."
}
```

如果 LLM 输出解析失败，应回退为 `needs_review`，不能直接判 badcase。

## 11. 自动失败模式发现

taxonomy 不采用人工预设，而采用自动发现。

### 11.1 失败摘要生成

对每个 `badcase_high_confidence` 和高风险 `needs_review` 样本生成失败摘要：

```text
入口模式：recommendation_entry
实际流程：直接调用客户痛点工具后生成大纲
关键失败：第2步工具返回无结果，第3步继续调用新工具，未向用户确认
证据规则：tool_failure_continue_without_user_confirmation
最终影响：要素收集未达目标，大纲依据不足
```

### 11.2 聚类方式

可使用两类表示进行聚类：

- 结构化风险向量：规则命中、风险分、入口模式、失败步骤位置。
- 文本摘要 embedding：失败摘要、关键 result、工具事件摘要。

聚类方法可选择：

- HDBSCAN。
- KMeans。
- Agglomerative Clustering。
- 基于相似度阈值的增量聚类。

### 11.3 自动命名

每个 cluster 自动抽取：

- 高频规则。
- 高频任务阶段。
- 代表样本。
- 共同失败摘要。

然后自动生成名称，例如：

- `cluster_01: 工具失败后继续推进流程`
- `cluster_02: 未经要素澄清直接生成大纲`
- `cluster_03: 重复调用同一推荐工具`
- `cluster_04: 要素推荐缺失但进入大纲生成`
- `cluster_05: 最终大纲与客户交流目标弱相关`

这些名称是结果，不是人工预设 taxonomy。

## 12. 输出格式建议

每条 trace 输出：

```json
{
  "name": "case_001.json",
  "label": "badcase_high_confidence",
  "confidence": 0.93,
  "entry_mode": "recommendation_entry",
  "decision_stage": "tool_compliance_rule",
  "risk_score": 0.91,
  "risk_factors": [
    "tool_failure_continue_without_user_confirmation",
    "self_repeated_tool_call"
  ],
  "evidence": [
    "第3步工具返回无结果",
    "第4步未向用户确认，直接调用新工具",
    "第5步重复调用同一工具且参数相同"
  ],
  "failure_summary": "工具未达目标后继续推进，并重复调用工具",
  "auto_failure_cluster": "cluster_01"
}
```

批量报告输出：

- 总样本数。
- `badcase_high_confidence` 数量和比例。
- `needs_review` 数量和比例。
- `goodcase_likely` 数量和比例。
- 各风险规则命中次数。
- 自动 failure cluster 分布。
- 每个 cluster 的代表样本。
- 建议抽检样本列表。

## 13. 无标注场景评估方法

无标注条件下不能直接计算真实 precision 和 recall，因此需要替代评估。

### 13.1 precision 估计

对 `badcase_high_confidence` 抽样人工核查：

- 每个 failure cluster 至少抽 3-5 条。
- 对高频规则单独抽样。
- 对 LLM 确认 badcase 和纯规则 badcase 分开估计 precision。

估计指标：

```text
estimated_precision = 抽样中真实 badcase 数 / 抽样总数
```

### 13.2 recall 间接监控

无标注 recall 无法直接计算，可用以下指标间接观察：

- high-confidence coverage：高置信 badcase 占总样本比例。
- review coverage：进入 `needs_review` 的比例。
- cluster diversity：自动失败 cluster 数量。
- repeated-risk capture：重复工具调用风险覆盖率。
- tool-failure capture：工具失败风险覆盖率。

如果 high-confidence coverage 极低，但 `needs_review` 很高，说明规则过于保守，需要增加 LLM 复核或放宽中等风险进入复核的阈值，而不是直接放宽 badcase 判定阈值。

### 13.3 误杀风险检查

precision 优先时，最重要的是检查误杀。

建议重点抽查：

- 入口为 `outline_entry` 但被判流程跳步的样本。
- 工具重复调用但可能由用户触发的样本。
- 工具失败后继续执行但中间可能已有用户确认的样本。
- 大纲较短但实际满足用户要求的样本。

这些样本如果误杀多，应调整规则到 `needs_review`。

## 14. 推荐落地步骤

第一步：实现 trace 结构化摘要。

- 提取 query、任务序列、工具序列、结果序列、最终输出。
- 识别空结果、失败结果、重复工具、循环模式。

第二步：实现入口模式识别。

- 先用关键词和简单语义规则。
- 无法识别时输出 `unknown_entry`。

第三步：实现高置信规则。

- 基础结构异常。
- 工具失败后继续执行新工具。
- 自行重复调用工具。
- 严重流程跳步。
- 最终大纲缺失。

第四步：实现风险评分。

- 将结构、流程、工具、结果、语义、确认六类风险合成 `risk_score`。
- 只把高分且命中强规则的样本判为 `badcase_high_confidence`。

第五步：实现 LLM 复核。

- 只处理 `needs_review` 或高风险未命中强规则样本。
- LLM 输出解析失败时保持 `needs_review`。

第六步：实现自动 failure cluster。

- 基于失败摘要和风险向量聚类。
- 自动生成 cluster 名称、代表样本和分布报告。

第七步：做抽样核查。

- 用少量人工抽检估计 precision。
- 根据误杀样本把规则从“直接 badcase”降级到 `needs_review`。

## 15. 最终建议

针对该高层客户交流助手，无标注场景的最佳方案不是“尽可能多地找 badcase”，而是“只把违反硬流程约束或强质量约束的 trace 判为高置信 badcase”。

最值得优先实现的高 precision 规则是：

1. 工具执行未达目标后，未展示结果和询问用户，直接继续调用新工具。
2. 无用户新输入或确认时，自行重复调用同一工具。
3. 用户直接要求大纲以外的场景中，严重跳过必要阶段。
4. 进入大纲生成阶段后，最终没有有效大纲。
5. trace 结构缺失或所有结果为空。

其它弱信号，例如任务顺序轻微变化、大纲较短、推荐内容泛化、入口模式不确定，不应直接判 badcase，应进入 `needs_review`。

该方案可以在没有标注数据的条件下优先保证 precision，同时通过 `needs_review`、风险聚类和抽样核查逐步补足 recall。
