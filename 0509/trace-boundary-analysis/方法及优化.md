# 方法及优化

本文档只记录“基于规则”的 trace 定界实验。当前阶段先不讨论特征法和 LLM 实现，目标是通过多轮实验找出哪些 trace 信息真正有利于制定高 precision 的 badcase 规则。

## 1. 当前实验目标

主要目标：

- 优先提升 `badcase` precision。
- 同时记录 `badcase` recall，避免规则过严后漏掉大量 badcase。
- 每条 badcase 判定必须能对应到明确规则和证据。
- 规则不要硬编码在 Python 里，优先写入 `rules.md`，由 `scripts/classify_rule.py` 读取。

## 2. 当前规则机制

规则配置文件：

- `rules.md`

规则执行脚本：

- `scripts/classify_rule.py`

当前设计原则：

- `rules.md` 的第一个区域用于向用户提问，收集业务规则。
- 用户补充后，把可执行信息沉淀到 `rules.md` 的规则配置区。
- Python 代码只负责读取配置并执行通用规则逻辑。
- 如果没有补充“什么任务代表成功完成”，规则法不会默认要求 trace 必须出现“生成大纲”。
- 当前规则法以高置信 badcase 拦截为主：只有命中明确 badcase 规则时才判为 `badcase`，否则暂判为 `goodcase`。

## 3. 当前已支持的通用规则

这些规则已在 `rules.md` 的配置区中描述，并由 `classify_rule.py` 读取。

### 3.1 plan_list 结构异常

如果 `plan_list` 缺失或不是 list，判为 `badcase`。

原因：

- trace 结构不可用，无法判断 Agent 是否按任务执行。
- 误杀风险较低。

### 3.2 plan_list 为空且没有最终结果

如果 `plan_list` 为空，并且配置的最终结果字段也都没有有效内容，判为 `badcase`。

原因：

- 没有任务过程，也没有可用最终答复。

### 3.3 所有步骤结果为空

如果 `plan_list` 中存在步骤，但所有步骤的 `result` 都为空，判为 `badcase`。

原因：

- Agent 没有产生有效中间结果。

### 3.4 工具失败后继续执行

如果某一步 `result` 命中失败标记，且后续继续出现新的 command，判为 `badcase`。

原因：

- 在通用 Agent trace 中，工具失败后继续推进通常意味着没有正确处理失败。
- 这条规则可能需要业务确认：如果某些工具允许失败后继续执行，应在后续实验中增加豁免。

### 3.5 重复或循环任务

如果连续任务模式重复达到配置阈值，判为 `badcase`。

当前建议：

- 为了优先 precision，默认阈值先设为 3。
- 任务签名默认包含 `task_name`、`command.name` 和 `command.args`，避免同一工具不同参数被误判为重复。

### 3.6 可选成功任务规则

`rules.md` 支持配置 `success_task_keywords`。如果未来你确认某些任务代表 trace 完成目标，可以填入这些关键词。

当前状态：

- 默认不配置任何成功任务关键词。
- 因此规则法不会因为“没有生成大纲”直接判 badcase。

## 6. 下一步分析重点

你运行实验后，优先补充以下信息：

- 哪些 FP 是因为“合理重复调用”导致的。
- 哪些 FP 是因为“工具失败后允许继续执行”导致的。
- 哪些 FN 看起来有明显失败信号，但当前规则没有抓到。
- 哪些 FN 需要业务知识才能判断，比如“应该生成大纲但没有生成”。

我会根据这些结果更新 `rules.md` 的问题区和规则配置区，再决定是否需要修改 `classify_rule.py` 的通用执行逻辑。



# SKILL流程

1、无trace和任务先验知识的通用规则，classify_rule.py可直接运行；

2、用户填写简单任务描述；Agent基于描述生成：

（1）基于已知字段的的规则，更新规则2和classify_rule.py --- prompt引导

（2）需要进一步澄清的问题，更新配置区问题

3、根据配置区信息生成新的规则并更新classify_rule.py

4、根据实验反馈跟新配置区问题

## 7. 实验记录：ensemble_sweep_20260511_172039

日期：2026-05-11

输入结果文件：
- `scripts/ensemble_sweep_20260511_172039.md`

实验目的：
- 检查 `rule + unsupervised` 的 OR/AND 组合是否能在保持高 `badcase precision` 的同时提升 `recall`。

主要结论：
- `ensemble_rule_unsupervised_and` 的最优结果为 precision 1.0000、recall 0.1724、TP=5、FP=0、FN=24、TN=80，基本等价于高置信规则法，recall 没有提升。
- `ensemble_rule_unsupervised_or` 的高 precision 折中结果为 precision 0.8750、recall 0.4828、TP=14、FP=2、FN=15、TN=78，和单独 `unsupervised` 的最优高 precision 前沿一致。
- 因此当前组合策略没有提供新增有效边界，说明 rule 命中的 badcase 已经被该参数区间下的 unsupervised 覆盖，继续调 OR/AND 组合收益较低。

本轮方法更新：
- 将 `unsupervised` 默认参数调整为当前高 precision 折中点：`threshold=0.55`、`bad_risk_threshold=0.45`、`bad_risk_weight=0.60`、`centrality_weight=0.20`。
- 新增 `--error-analysis-output`，用于固定参数后输出 FP/FN/TP 明细，下一轮优先分析 2 个 FP 和 15 个 FN，而不是继续只看总指标。

下一轮建议：
- 运行默认 `unsupervised` 并输出误差分析：

```powershell
python .\scripts\run_trace_analysis.py <trace_json_or_dir> --metadata <metadata.csv> --strategy unsupervised --error-analysis-output unsupervised_error_analysis.md
```

- 如果 2 个 FP 属于“合理重复/合理空结果/合理未生成大纲”，应增加保护性 goodcase 特征或提高对应风险阈值。
- 如果 15 个 FN 中存在明显失败模式，应优先把可解释、低误杀的模式沉淀到 `rules.md` 或新增 bad-risk 子特征，而不是扩大 OR ensemble。
