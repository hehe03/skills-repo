# 方法及优化

本文档只记录“基于规则”的 trace 定界实验。当前阶段先不讨论特征法和 LLM 实现，目标是通过多轮实验找出哪些 trace 信息真正有利于制定高 precision 的 badcase 规则。

## 1. 当前实验目标

主要目标：

- 优先提升 `badcase` precision。
- 同时记录 `badcase` recall，避免规则过严后漏掉大量 badcase。
- 每条 badcase 判定必须能对应到明确规则和证据。
- 规则不要硬编码在 Python 里，优先写入 `rules.md`，由 `scripts/classify_rule.py` 读取。

## 2. 当前规则机制

规则配置文件：

- `rules.md`

规则执行脚本：

- `scripts/classify_rule.py`

当前设计原则：

- `rules.md` 的第一个区域用于向用户提问，收集业务规则。
- 用户补充后，把可执行信息沉淀到 `rules.md` 的规则配置区。
- Python 代码只负责读取配置并执行通用规则逻辑。
- 如果没有补充“什么任务代表成功完成”，规则法不会默认要求 trace 必须出现“生成大纲”。
- 当前规则法以高置信 badcase 拦截为主：只有命中明确 badcase 规则时才判为 `badcase`，否则暂判为 `goodcase`。

## 3. 当前已支持的通用规则

这些规则已在 `rules.md` 的配置区中描述，并由 `classify_rule.py` 读取。

### 3.1 plan_list 结构异常

如果 `plan_list` 缺失或不是 list，判为 `badcase`。

原因：

- trace 结构不可用，无法判断 Agent 是否按任务执行。
- 误杀风险较低。

### 3.2 plan_list 为空且没有最终结果

如果 `plan_list` 为空，并且配置的最终结果字段也都没有有效内容，判为 `badcase`。

原因：

- 没有任务过程，也没有可用最终答复。

### 3.3 所有步骤结果为空

如果 `plan_list` 中存在步骤，但所有步骤的 `result` 都为空，判为 `badcase`。

原因：

- Agent 没有产生有效中间结果。

### 3.4 工具失败后继续执行

如果某一步 `result` 命中失败标记，且后续继续出现新的 command，判为 `badcase`。

原因：

- 在通用 Agent trace 中，工具失败后继续推进通常意味着没有正确处理失败。
- 这条规则可能需要业务确认：如果某些工具允许失败后继续执行，应在后续实验中增加豁免。

### 3.5 重复或循环任务

如果连续任务模式重复达到配置阈值，判为 `badcase`。

当前建议：

- 为了优先 precision，默认阈值先设为 3。
- 任务签名默认包含 `task_name`、`command.name` 和 `command.args`，避免同一工具不同参数被误判为重复。

### 3.6 可选成功任务规则

`rules.md` 支持配置 `success_task_keywords`。如果未来你确认某些任务代表 trace 完成目标，可以填入这些关键词。

当前状态：

- 默认不配置任何成功任务关键词。
- 因此规则法不会因为“没有生成大纲”直接判 badcase。

## 6. 下一步分析重点

你运行实验后，优先补充以下信息：

- 哪些 FP 是因为“合理重复调用”导致的。
- 哪些 FP 是因为“工具失败后允许继续执行”导致的。
- 哪些 FN 看起来有明显失败信号，但当前规则没有抓到。
- 哪些 FN 需要业务知识才能判断，比如“应该生成大纲但没有生成”。

我会根据这些结果更新 `rules.md` 的问题区和规则配置区，再决定是否需要修改 `classify_rule.py` 的通用执行逻辑。
