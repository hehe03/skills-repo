# skill-ops 洞察报告

## 1. 本项目面对的不是“单纯调参”，而是“规则、实现、标注”三者共同优化

`shortage_analyze` 这类问题的难点，不在于单条规则是否能写出来，而在于以下三件事同时存在：

1. 业务规则来自人工总结，天然可能遗漏边界条件；
2. 规则文档与代码实现之间可能漂移；
3. 人工标注本身并不一定稳定，甚至可能和字段证据冲突。

因此，`skill-ops` 不能只做“改代码提分”，而要做“证据驱动的规则运营”。

## 2. 业界做法与可借鉴点

### 2.1 把规则当成可管理的弱监督来源

[Snorkel DryBell](https://arxiv.org/abs/1812.00417) 讨论了如何把组织内已有规则、启发式和知识源当作可管理的弱监督信号。对本项目最有价值的启发有两点：

1. 规则不是一次性产物，而是持续迭代的“标注函数”；
2. 优化时要关注规则覆盖面、规则冲突和规则组合后的整体收益。

映射到 `skill-ops`，意味着我们不把 `rules.md` 当静态文档，而是把每条规则都当成一个可观测、可回放、可评估的对象。

### 2.2 先定位疑似脏标注，再决定是否改规则

[Confident Learning](https://arxiv.org/abs/1911.00068) 的核心思想是：不要默认训练数据标签都可靠，应优先识别“最像错标”的样本。它虽然通常用于概率模型，但思路同样适用于规则型系统：

1. 若规则证据强且稳定，而标注与之冲突，优先把该样本列为“疑似错标”；
2. 若规则证据弱、实现简化或边界不清，再考虑改规则。

这正是本项目里“规则与标注冲突时，优先考虑修规则，但保留错标清单”的方法来源。

### 2.3 数据文档与证据保留是长期可维护性的前提

[Datasheets for Datasets](https://arxiv.org/abs/1803.09010) 和 [Data and its (dis)contents](https://pmc.ncbi.nlm.nih.gov/articles/PMC8600147/) 都强调：如果数据来源、字段含义、使用方式和局限没有被记录，后续优化会反复掉进同样的坑。

对 `skill-ops` 的直接启发是：

1. 除了规则文档，还要保留 trace 和案例诊断报告；
2. 评估脚本要能直接输出“哪里错、为什么错、错得是否稳定”；
3. 不能只产出一个更高分的脚本，而要产出一套能持续复盘的数据与规则文档。

### 2.4 高风险场景里，最贵的不是错一次，而是错了还无法追责

[Data Cascades in High-Stakes AI](https://research.google/pubs/everyone-wants-to-do-the-model-work-not-the-data-work-data-cascades-in-high-stakes-ai/) 提醒我们，数据问题往往是延迟暴露、层层叠加的。对规则型 Agent 来说，缺少 trace 就是最典型的“数据级技术债”。

因此，本项目把“先补 trace”作为正式优化前置步骤，而不是可选项。

## 3. 本项目采用的落地策略

结合以上资料，`skill-ops` 最终采用以下策略：

1. 先补 trace，再谈优化。
2. 用一致性率、漏报/误报分布做基线，不凭直觉改规则。
3. 对每个候选改动做反事实评估，只接受“能解释一类样本”的规则调整。
4. 将剩余错例显式区分为：
   - 规则实现缺口
   - 规则定义不清
   - 标注疑似错误
   - 数据不足

## 4. 对 shortage_analyze 案例的直接启发

本案例最值得优先处理的不是所有差异，而是“有明确证据链的差异”：

1. `替代交付异常` 原实现把“有组合替代库存字段”近似成“组合替代可满足”，这属于明确实现缺口；
2. `基线异常` 缺少“推荐基线显著高于原始基线，但历史需求为零”的边界条件，这属于文档可补充、规则可泛化的缺口；
3. 剩余大量 `替代交付异常` 标签发生在 `组合替代关系=[]` 的样本上，和当前规则文档直接冲突，更像标注侧不一致，而不是继续改规则的理由。

## 5. 结论

`skill-ops` 不是一个“自动刷分”工具，而是一套证据化运营规则型 skill 的方法。它的价值在于：

1. 把规则优化从“经验修改”变成“可追踪实验”；
2. 把标注问题从“口头怀疑”变成“结构化疑点”；
3. 把一次性的案例修复沉淀成可复用流程。
