# skill-ops 算法架构说明

## 1. 总体目标

`skill-ops` 面向的是一类“规则型 Agent skill 优化”问题：

1. 已有工作 skill；
2. 已有测试数据和部分人工标注；
3. 工作 skill 预测结果与标注结果不一致；
4. 希望在不硬编码训练集答案的前提下，提升一致性率，并区分规则问题与标注问题。

当前 `skill-ops` 的算法架构可以拆成三层：

- Trace 层：把规则执行过程显式化；
- 诊断层：把错例结构化分型；
- 优化层：基于反事实影响做保守规则优化。

## 2. Trace 层算法

### 2.1 逐规则执行追踪

核心思想是把工作 skill 中的每条规则都拆成：

1. 条件集合；
2. 字段证据；
3. 匹配结论；
4. 规则解释。

这相当于一个轻量级的 `rule execution trace`。它不是黑盒预测，而是显式记录每条规则“为什么命中 / 为什么没命中”。

### 2.2 字段归一化与结构化解析

为了让 trace 可计算，当前做了三类基础处理：

1. 数值安全比较：处理空值、字符串数值、NaN；
2. 单位归一化：将 `d/h` 时长统一转成天；
3. 结构化对象解析：将组合替代库存、组合替代需求解析成 SKU-数量对。

这些处理本质上是规则引擎前的数据标准化算法。

### 2.3 细粒度组合核算

在 `shortage_analyze` 案例里，`替代交付异常` 的判断不再停留在“有无组合库存字段”层面，而是进一步做：

1. SKU 级聚合；
2. 需求与库存逐项比较；
3. 输出不足 SKU 列表。

这属于一个简单但重要的 `constraint checking` 算法：逐约束验证组合是否满足需求。

## 3. 诊断层算法

### 3.1 集合式标签比较

对多标签结果，`skill-ops` 当前不做字符串逐字比较，而是把 `answer` 和 `prediction` 都映射成标签集合，计算：

1. 精确一致；
2. 漏报标签；
3. 误报标签。

这是一个多标签集合差分算法。

### 3.2 错例分布统计

在标签集合比较基础上，继续统计：

1. 每个标签的漏报次数；
2. 每个标签的误报次数；
3. 主要冲突标签。

这一步属于误差分解和分布分析。

### 3.3 重复样本冲突检测

忽略 `references`、`answer`、`L2分类结果` 后，对剩余特征做分组，识别：

1. 相同输入；
2. 不同标注。

这是一个近似的数据一致性检查算法，用于发现疑似错标或标注规范漂移。

### 3.4 规则/标注反事实对照分析

对于“某类标注与规则文档冲突”的现象，当前使用双方案反事实评估：

1. 若修改规则，会修复多少样本、破坏多少样本；
2. 若修改标注，会修复多少样本、破坏多少样本。

然后比较两种方案的净收益与副作用。

这一步本质上是一个局部反事实评估算法，不是因果识别，但足够支持规则运营决策。

## 4. 优化层算法

### 4.1 规则补全

当 trace 显示“规则文档存在，但代码实现简化或遗漏”时，采取规则补全。

在 `shortage_analyze` 中，这体现在：

1. `替代交付异常` 从粗粒度判断升级为 SKU 级约束核算；
2. `基线异常` 增加稳定边界条件。

### 4.2 保守式规则优化

当前 `skill-ops` 不做全局自动搜索，而采用保守策略：

1. 只修改有证据链支撑的规则；
2. 优先接受副作用较小的规则变更；
3. 若冲突更像标注问题，则不为训练集放宽规则。

这可以看作一种“人工主导 + 反事实验证”的启发式规则学习。

## 5. 当前架构在 shortage_analyze 上的实际效果

在 `shortage_analyze` 案例中，当前架构成功完成了三件事：

1. 生成结构化 trace；
2. 修复明确规则实现缺口；
3. 识别出 `组合替代关系=[]` 但标注为 `替代交付异常` 更像标注冲突，而不是规则应继续放宽。

## 6. 当前架构的局限

当前版本仍然偏“分析型工具链”，主要局限是：

1. 候选规则仍主要靠人工提出；
2. 标注疑点排序还不够系统；
3. 对高频冲突模式的发现能力依然偏手工；
4. 反事实评估多针对单个假设，而不是自动搜索一批候选假设。

## 7. 下一步算法演进方向

后续更适合增强的方向包括：

1. 候选规则自动评估；
2. 标注疑点自动打分；
3. 高频冲突模式自动挖掘；
4. 多规则联合优化而不是单规则局部修改；
5. 阈值敏感性分析与鲁棒性评估。

这些增强内容将放到后续的 `skill-ops-v1` 中。
