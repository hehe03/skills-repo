# skill-ops

`skill-ops` 用于优化“工作 skill”。这里的工作 skill 可以是一段 prompt、一个脚本、一个完整的 Agent 操作说明，核心目标是提升预测结果与人工标注的一致性，同时区分“规则缺陷”和“标注缺陷”。

## 适用输入

一个可优化案例通常至少包含以下内容：

1. 工作 skill 说明文档，例如 `SKILL.md`、规则文档或 SOP；
2. 可执行实现，例如脚本、prompt 模板或工作流配置；
3. 测试数据；
4. 人工标注列和工作 skill 预测列，或可重新跑出预测列的输入数据。

## 核心原则

1. 先保留证据，再改规则：没有 trace 时，优先补 trace。
2. 优先修正规则缺陷，不根据训练集硬编码答案。
3. 对剩余差异做分诊：规则错误、标注疑似错误、信息不足。
4. 所有结论都要落到可复盘证据：命中的规则、字段值、反事实影响。
5. 当标注与规则冲突时，不能只给结论，必须比较“改规则”和“改标注”各会带来什么后果。

## 标准流程

### 1. 建立基线

1. 读取工作 skill 文档、规则文档和实现脚本；
2. 确认数据列、标签列、预测列；
3. 计算基线一致性率、错例总数、每个标签的漏报/误报。

### 2. 检查并补齐 trace

若工作 skill 缺少逐规则证据输出，需要先补一个 `generate_trace.py`，保证每行数据至少能输出：

- 命中的规则名；
- 每条规则的布尔条件结果；
- 关键字段原值；
- 匹配/不匹配原因。

建议输出为 `jsonl`，每行一个样本，便于自动分析。

### 3. 错例分诊

根据 trace 和训练集，将错例优先分成以下几类：

1. 规则实现缺口：文档有规则，但代码实现简化或遗漏；
2. 规则定义冲突：不同规则命中逻辑相互掩盖或边界不清；
3. 标注疑似错误：相同数据的标签互相冲突；
4. 疑似规则或标注错误：字段证据与人工标签明显冲突；
5. 数据缺失：当前字段不足以支持稳定判断。

### 4. 反事实评估

对每个候选改动先做“如果修改，会影响哪些样本”的评估，只接受以下改动：

1. 能解释一类样本，而不是只修单点样本；
2. 证据链可复述；
3. 不引入明显的新误报。

当发现“规则文档与标注冲突”时，至少要输出两种方案的对照：

1. 如果修改规则，一致性率会怎么变化，新增修复和新增破坏各有多少；
2. 如果修改标注，一致性率会怎么变化，新增修复和新增破坏各有多少；
3. 基于上述反事实比较，推断更可能是哪种情况。

### 5. 形成交付物

`skill-ops` 的标准输出至少包括：

1. 一份案例诊断报告：基线、问题分型、改动、收益、剩余风险；
2. 一套可复用脚本：基线评估、trace 分析、反事实对照分析、案例报告生成；
3. 更新后的工作 skill 规则/脚本，以及 trace 生成能力。

## 在案例中如何落地

以 `shortage_analyze` 为例，本次实践采用如下动作：

1. 为 `shortage_analyze/scripts/analyze_shortage.py` 增加结构化 trace；
2. 新增 `shortage_analyze/scripts/generate_trace.py`；
3. 运行 `generate_trace.py`，得到每个样本的规则命中轨迹；
4. 分析样本 trace，对错例进行分诊；
5. 对“规则冲突 vs 标注冲突”样本运行反事实对照分析；
6. 基于错例分析，尝试新增、删除或修改规则，使得训练集一致性率提升；
7. 若一致性率提升，输出优化报告和优化后的 `shortage_analyze` 技能，文件保存到 `./opt/shortage_analyze`，原 `shortage_analyze` 保持不变。

## 推荐工具

- `skill-ops/scripts/evaluate_case.py`
  作用：计算一致性率、漏报/误报分布、疑似重复冲突组。
- `skill-ops/scripts/analyze_counterfactuals.py`
  作用：针对规则和标注冲突样本，比较“改规则”和“改标注”的后果。

## 注意事项

1. 文档统一用 UTF-8 编码，兼容 Windows。
2. 若预测列来自历史运行结果，不要直接信任，优先基于当前脚本重跑。
3. 不要为了“刷分”而牺牲泛化能力。
